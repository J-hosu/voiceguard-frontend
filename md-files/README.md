# VoiceGuard AI — 프론트엔드(앱) 개발 문서

> 이 문서 묶음은 **바이브코딩(AI 코드 생성)용 컨텍스트 파일**입니다.
> AI 코딩 도구(Claude Code / Cursor 등)에 통째로 넣거나, 작업 중인 화면에 맞는 파일만 골라 넣어 사용하세요.
> 담당 범위는 **앱(프론트엔드)만**입니다. 백엔드는 별도 담당이며, 프론트에서 필요하지만 백엔드에 아직 없는 것은 `05_BACKEND_REQUESTS.md`로 분리해 두었습니다.
>
> ⚙️ **기술 스택 변경(확정): Flutter → React Native + Expo (TypeScript).** 사유·실행법은 `01_TECH_STACK.md`와 프로젝트 루트 `변경사항_보고.txt` 참조. 이 문서들의 API/화면/디자인/백엔드요청 **내용 자체는 그대로 유효**하며, 일부 Dart 코드 예시는 참고용(실제 구현은 프론트 `src/`).

## ⚠️ 먼저 읽어야 할 핵심 사실

작업 시작 전에 아래 3가지를 반드시 이해하고 있어야 합니다. 기존 기획 문서와 실제 백엔드가 서로 다릅니다.

1. **API 기준은 `API_SPEC.md`(실제 구현본)입니다.**
   기능명세서 §6과 프론트엔드 스펙 docx §3.2에 나오는 `POST /api/calls/upload`, `/transcribe`, `/chunks`, `GET /api/calls/{id}/result` 같은 엔드포인트는 **실제 백엔드에 존재하지 않습니다.** 그 문서들은 초안이고, 실제 백엔드는 다르게 구현됐습니다. 코딩 시 이 초안 엔드포인트를 절대 호출하지 마세요. → 자세한 내용 `02_API_INTEGRATION.md`

2. **백엔드는 STT(음성→텍스트)를 하지 않습니다.**
   실시간 분석용 WebSocket은 **이미 텍스트로 변환된 발화**를 받습니다(`{"type":"message","content":"검찰입니다...","turn_index":1}`). 따라서 **음성→텍스트 변환은 앱이 온디바이스로 직접 수행**해야 합니다. 이것이 프레임워크·패키지 선택과 실시간 화면 구조를 결정합니다. → `01_TECH_STACK.md`, `05_BACKEND_REQUESTS.md`

3. **"음성 파일 업로드 분석"(피그마 화면 3)은 [확정] 온디바이스 STT로 우회합니다.**
   파일 업로드/파일 STT 엔드포인트가 백엔드에 없으므로, 앱이 파일을 온디바이스 STT로 전사해 WS 파이프라인에 흘려보냅니다(백엔드 요청 없이 진행). 구현 리스크/폴백은 `03_SCREENS.md` 화면 3, `05_BACKEND_REQUESTS.md` 항목 1 참조.

## 프로젝트 한 줄 요약

통화 중(또는 녹음 파일)의 발화를 실시간 텍스트로 바꾸고, 보이스피싱 위험도를 계산해 일정 기준 이상이면 **"왜 위험한지" 근거와 함께** 즉시 경고하는 안드로이드 앱.

상용 앱이 "경고 중심"이라면, 이 앱은 **설명 중심(Explainable) UX**가 차별점입니다.

## MVP 범위 (5일 데모 기준)

| 구분 | 내용 |
| --- | --- |
| 타겟 | 안드로이드 단일 (Galaxy S21+ 이상, 테스트: S24 Ultra) |
| 필수 시나리오 | ① 실시간 마이크 입력 → 온디바이스 STT → WS 분석 → 경고 |
| 후순위 시나리오 | ② 녹음 파일 분석(백엔드 제약으로 우회 구현 필요) |
| 배포 | APK 사이드로딩(필수) / Play 내부 테스트(선택) |
| 실제 통화 앱 연동 | **불가** (Android 정책). 스피커폰 + 마이크 기반 데모로 대체 (기능명세서 §13) |
